# Решение: выбрать Kotlin для разработки

## Контекст

При запуске проекта ComminusmPlugin потребовалось выбрать язык программирования для реализации логики плагина. В экосистеме Minecraft Bukkit/Spigot традиционно преобладал Java, однако современные тренды указывали на необходимость обновления технологического стека.

## Решение

Выбран **Kotlin** как основной язык реализации плагина.

## Причины

1. **Краткость и читаемость** — Kotlin позволяет表达 сложную логику в 30-50% меньше строк кода по сравнению с Java

2. **Безопасность типов** — null-safety и система типов Kotlin снижают вероятность runtime-ошибок в игровом коде

3. **Интероперабельность** — Kotlin полностью совместим с Java API PaperSpigot, не требуются обертки

4. **Современный стек** — поддержка современных фич языка (coroutines, extension functions) упрощает разработку

5. **Модная тренд** — большинство новых Minecraft-плагинов на Java переносятся на Kotlin, что упрощает найм разработчиков

## Последствия

### Положительные
- ✅ Упрощенная синтаксис (`override fun onEnable()` вместо `public void onEnable()`)
- ✅ Автоматическая генерация геттеров/сеттеров
- ✅ Extension functions для упрощения API PaperSpigot
- ✅ Легче поддерживать код при расширении функциональности

### Отрицательные
- ⚠️ Требуется дополнительная настройка сборки (Gradle плагин kotlin("jvm"))
- ⚠️ Немного больший размер JAR-файла из-за stdlib
- ⚠️ Некоторые разработчики могут не знать Kotlin (но это быстро компенсируется)

## Альтернативы

1. **Java** — традиционный выбор, но потеря современных фич
2. **Scala** — более сложная кривая обучения, редкий стек
3. **Groovy** — устаревший стек, нет перспектив развития
