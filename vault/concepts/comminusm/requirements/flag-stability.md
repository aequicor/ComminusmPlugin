# Requirements: Flag Stability

**Module:** comminusm
**Status:** Draft
**Date:** 06.05.2026
**Author:** @BusinessAnalyst

---

## 1. Overview

### Business Context

В текущей реализации флаги приватов (белый баннер ордера, красный баннер трудового фронта) являются уязвимыми. При разрушении опорного блока (игроком, взрывом, водой, лавой, поршнем) баннер выпадает как предмет. Это создаёт два критических риска:

1. **Дюп флагов** — игрок ломает опору → подбирает флаг → запрашивает новый через `/party` → получает второй флаг. Повторяемо.
2. **Потеря привата без ведома владельца** — взрыв крипера/TNT разрушает опорный блок → флаг выпадает → привязанная зона продолжает существовать в БД, но флаг и опора исчезли. Игрок не знает, что его приват скомпрометирован.

Существующая защита покрывает только прямое разрушение блока баннера (`BlockBreakEvent` + `ExplosionListener`), но **не покрывает разрушение опорного блока** взрывом, водой, поршнем, лавой и т.д., а также **не предотвращает дюп** через цепочку «сломал → подобрал → запросил новый».

**Цель:** сделать флаги неуничтожимыми в игровом мире, исключить дюп, добавить визуальную индикацию принадлежности.

### Feature Value

| Кому | Ценность |
|------|----------|
| Владельцу привата | Флаг невозможно случайно или злонамеренно уничтожить. Приват стабилен. |
| Другим игрокам | Видят имя владельца над флагом — ясная принадлежность территории. |
| Администратору сервера | Нет жалоб на пропажу приватов. Нет расследований дюпов флагов. |
| Разработчику | Упрощается модель: флаг — статичный объект мира, а не предмет инвентаря. |

| AC-24 | US-04, NFR-02 | Игрок активирует флаг, сервер крашится после замены опоры (бедрок), но до создания armor stand | Сервер перезапускается, загружается чанк | ChunkLoadEvent различает случаи по PDC чанка: <br>• **Опорный блок BEDROCK/OBSIDIAN присутствует, ключ armor stand в PDC отсутствует** → это **новый флаг (крашнулся при активации)**. Система создаёт armor stand и сохраняет его UUID в PDC (full repair, завершение активации). <br>• **Опорный блок НЕ BEDROCK/OBSIDIAN, баннер присутствует** → это **старый флаг (до обновления)**. NFR-02: «не трогать» — система **ничего не делает** с legacy флагом. <br>- Ключевой discriminator: материал опорного блока. BEDROCK/OBSIDIAN = новый флаг (нужен repair); любой другой материал = старый флаг (пропустить). |
| AC-25 | US-03, FR-05 | Игрок-владелец сменил никнейм через Mojang | Следующая загрузка чанка или игрок входит на сервер | Armor stand не обновляется мгновенно (lazy refresh при загрузке чанка). При загрузке чанка система сверяет `nameTag` armor stand с `Bukkit.getOfflinePlayer(uuid).name`: если расходятся — обновляет. Между моментами (до следующей загрузки чанка) может отображаться устаревший никнейм — это приемлемо (A-01). Активное обновление при `PlayerJoinEvent` владельца — опционально, не обязательно. |

## 3. Functional Requirements

### FR-01: Indestructible Support Block on Activation

При активации флага ордера (`OrderFlagListener.onBlockPlace`) или флага фронта (`FrontFlagListener.onBlockPlace`) сервис должен:

1. Проверить, что над местом установки баннера есть **минимум 2 свободных блока** воздуха (Material.AIR или CAVE_AIR).
2. Если проверка не пройдена — отменить активацию, отправить игроку сообщение.
3. Заменить опорный блок (блок непосредственно под баннером) на **неуничтожимый материал** (конфигурируемый: `BEDROCK` по умолчанию, `OBSIDIAN` как опция).
4. Создать **невидимый armor stand** (`ArmorStand`, `setVisible(false)`, `setGravity(false)`, `setMarker(true)`) на 1 блок выше баннера.
5. Установить на armor stand **кастомное имя** (`setCustomNameVisible(true)`) с именем владельца (никнейм игрока).
6. Сохранить UUID созданного armor stand в **PDC чанка** (`Chunk.getPersistentDataContainer`) по ключу `comminusm:armorstand/{orderId}` или `comminusm:armorstand/{frontId}`. PDC чанка выбран, потому что: (а) он персистентен и сохраняется вместе с миром, (б) не требует миграции схемы БД, (в) автоматически мигрирует при переносе мира. При потере PDC (сброс чанка, перенос на сервер без плагина) — используется fallback-поиск entity по координатам.

### FR-02: Immutable Protection for All Destruction Paths

Система должна блокировать разрушение/перемещение опорного блока флага через **все** возможные механизмы:

| Механизм | Событие/механизм | Действие |
|----------|------------------|----------|
| Игрок ломает (survival) | `BlockBreakEvent` | Cancel. Сообщение игроку (отличать своего/чужого). Для своего ордера — GUI подтверждения удаления. |
| Игрок ломает (creative) | `BlockBreakEvent` | Cancel. Бедрок не ломается даже в creative (PaperSpigot 1.21 позволяет отменять). |
| Команда /setblock, /fill, /clone | Нет события (PaperSpigot 1.21 limitation) | Пассивная верификация при ChunkLoadEvent: если опорный блок подменён — восстановить BEDROCK/OBSIDIAN. Если опорный блок удалён командой — AC-23 (lazy repair). Логирование WARN. |
| Прямое разрушение админом (creative/setblock) | NFR-03 | Администратор может удалить флаг через сервис (API), но не через прямое разрушение. Любая прямая подмена опоры будет восстановлена при загрузке чанка. |
| Взрыв entity | `EntityExplodeEvent` | Удалить из `blockList()` опорный блок **и** блок баннера (блок над опорой). Баннеры имеют низкую blast resistance (~1) и уничтожаются даже слабыми взрывами. |
| Взрыв блока | `BlockExplodeEvent` | Удалить из `blockList()` опорный блок **и** блок баннера. |
| Wither-босс | `EntityExplodeEvent` | Бедрок неразрушим для Wither (vanilla-механика). При OBSIDIAN — удалить оба блока (опора + баннер) из `blockList()`. |
| Поток воды/лавы | `BlockFromToEvent` | Cancel, если flowing блок пытается занять позицию опорного блока **или** блок баннера. Баннер также должен быть защищён от смывания. |
| Поршень толкает | `BlockPistonExtendEvent` | Cancel, если опорный блок или блок баннера в списке двигаемых. |
| Поршень втягивает | `BlockPistonRetractEvent` | Cancel, если опорный блок или блок баннера в списке двигаемых. |
| Падающий блок (песок/гравий) | `EntityChangeBlockEvent` | Cancel, если целевой блок — опора флага **или** блок баннера. |
При этом **разрушение самого баннера напрямую** должно быть также недоступно (существующая защита в `BlockListener.onBlockBreak` сохраняется и дополняется).

### FR-03: No Flag Item Drop — Zero Tolerance

Ни при каких обстоятельствах флаг (WHITE_BANNER / RED_BANNER) не должен появляться в мире как `Item` entity (дроп) при разрушении опорного блока или самого баннера. Это исключает дюп.

- При удалении ордера через GUI (`FlagDeletionConfirmListener`): флаг не дропается (уже исправлено в DEF-User-04).
- При деактивации фронта владельцем: флаг добавляется в инвентарь (`giveOrNotify`), а не дропается. Если инвентарь полон — предупреждение, но флаг не теряется (можно получить новый через `/party`).
- При любом несанкционированном разрушении: событие отменяется, дропа нет.

### FR-04: Clean Removal on Delete / Deactivate

При удалении ордера (`OrderService.deleteByOwner`) или деактивации фронта (`WorkFrontService.deactivate`):

1. Удалить armor stand (entity) над флагом — искать по UUID из PDC чанка, fallback: поиск `ArmorStand` в радиусе 3 блоков от позиции флага.
2. Заменить неуничтожимый опорный блок на **MATERIAL_AIR**. Исходный материал опорного блока **не сохраняется**. Образовавшаяся дыра (воздух на месте бывшего флага) — приемлемое поведение: владелец привата должен застроить это место самостоятельно.
3. Удалить блок баннера (существующая логика).
4. Удалить ключ armor stand из PDC чанка.
5. Очистить кэш чанка (существующая логика).
6. Удалить запись из БД (существующая логика).

### FR-05: Owner Title on Armor Stand

- **Содержимое title:** никнейм игрока-владельца (plain text), получаемый через `Bukkit.getOfflinePlayer(uuid).name` на момент активации.
- **Формат:** настраиваемый префикс/постфикс через конфиг (по умолчанию: `«Ордер» — {playerName}` для ордера, `«Трудовой Фронт» — {playerName}` для фронта).
- **Видимость:** armor stand невидим (`setVisible(false)`), но имя видно (`setCustomNameVisible(true)`).
- **Поведение:** armor stand — маркерный (`setMarker(true)`), не имеет хитбокса, не взаимодействует с физикой.
- **Персистентность:** armor stand должен сохраняться между перезапусками сервера и перезагрузками чанков (устанавливается при создании; стандартное поведение entity).
- **Передача владения:** при изменении владельца ордера/фронта (через админскую передачу, если такой механизм существует) — имя на armor stand должно обновляться: (1) синхронно в момент передачи, если armor stand доступен; (2) при загрузке чанка (lazy refresh) — сверять имя на armor stand с актуальным владельцем из БД и обновлять при расхождении.

### FR-06: Configuration

Новые параметры в `PluginConfig` (`config.yml`):

| Параметр | Тип | По умолчанию | Описание |
|----------|-----|-------------|----------|
| `flag.supportBlockMaterial` | `String` | `BEDROCK` | Материал опорного блока (`BEDROCK` или `OBSIDIAN`) |
| `flag.minAirAbove` | `Int` | `2` | Минимум свободных блоков воздуха над баннером |
| `flag.titleFormat` | `String` | `§6{type} — §f{player}` | Формат title над флагом. `{type}` = «Ордер» / «Трудовой Фронт», `{player}` = никнейм |
| `flag.maxPerChunk` | `Int` | `50` | Максимальное количество флагов в одном чанке. Учитывает entity-лимит чанка (vanilla: ~5 типов entity на 16×16 секцию, armor stand — маркерный, потребляет 1 slot). Значение 50 — безопасный запас. |
| `flag.allowedWorlds` | `List<String>` | `["world"]` | Список миров (имён), в которых разрешена установка флагов. По умолчанию — только Overworld. |

### FR-07: Concurrent Activation Safety

При активации флага сервис должен гарантировать, что одна локация не получит два флага одновременно:

1. **Перед заменой опорного блока** — проверить, что целевая позиция не занята существующим флагом (поиск в кэше чанка).
2. **Синхронизация** — все операции с флагами в пределах одного чанка должны выполняться под `synchronized`-блоком по ключу чанка (`chunkKey = worldName + ":" + chunkX + ":" + chunkZ`), чтобы два вызова из разных потоков не создали гонку.
3. **Откат при ошибке создания armor stand** — если Bukkit API возвращает ошибку при `world.spawn(armorStand)`, опорный блок восстанавливается в AIR, активация отменяется. Игрок получает сообщение об ошибке.
4. **Краш-сервера между шагами** — при `ChunkLoadEvent` плагин проверяет консистентность: если обнаружен опорный блок флага без armor stand (в PDC чанка нет ключа armor stand) — создаётся armor stand (lazy repair). Альтернатива: осиротевший бедрок заменяется на AIR, владелец может переактивировать.

## 4. Non-Functional Requirements

### NFR-01: Performance

- Armor stand — лёгкая entity (маркерная, без физики). Создание/удаление при активации/деактивации — O(1). Не должно создавать задержек при установке флага.
- Проверка 2 блоков воздуха — 2 вызова `world.getBlockAt()`, O(1).
- Поиск опорного блока при защите от взрыва/поршня — уже существующая логика `isOrderFlag`/`isFrontFlag`, дополняется фильтрацией по материалу опорного блока.
- При загрузке мира/чанка с существующими armor stand'ами — накладных расходов нет (entity загружаются штатно).
- **Лимит entity на чанк:** vanilla-лимит составляет ~5 entity-типов на 16×16×16 секцию. Armor stand с `setMarker(true)` считается одним entity. При 50 флагах на чанк (значение по умолчанию для `flag.maxPerChunk`) остаётся значительный запас для других entity (мобы, item frames, etc.).

### NFR-02: Backward Compatibility

- Существующие флаги в мире, активированные до этого обновления, **не должны сломаться**. Плагин не должен требовать переустановки всех флагов.
- **Миграция:** применяется подход **«не трогать»** — существующие флаги **НЕ модифицируются** при старте плагина. Новое поведение (неуничтожимый опорный блок + armor stand) применяется **только к новым активациям флагов**.
- **Почему не lazy migration:** автоматическое достраивание защиты для старых флагов при загрузке чанка создаёт риск изменения мира без явного действия игрока. Подход «не трогать» гарантирует, что обновление плагина не меняет существующие структуры. Владельцы существующих приватов могут переактивировать флаг (удалить через GUI и установить заново), чтобы получить защиту.
- **Feature Value для владельцев существующих приватов:** достигается через документированную процедуру переактивации. Администратор сервера информирует игроков при обновлении плагина.
- Команда `/party` и GUI меню (`PartyMenu`, `OrderMenu`, `FrontMenu`) продолжают работать без изменений.

### NFR-03: Admin Override

- Администратор с правом `comminusm.admin` должен иметь возможность удалить опорный блок флага (например, через `/party admin`), но это должно выполняться контролируемо через сервис (не через прямое разрушение блока).
- Конкретный механизм админского удаления/восстановления флагов может быть реализован в будущем админском инструментарии; в рамках этого требования достаточно обеспечить, чтобы админ-разрушение не приводило к дюпу.

### NFR-04: Error Resilience

- Если armor stand не может быть создан (например, из-за ошибки Bukkit API), активация флага должна быть отменена, а опорный блок возвращён в исходное состояние.
- Если при удалении флага armor stand не найден в мире (уже убит/потерян), удаление опорного блока и записи БД должно продолжиться без ошибок (graceful degradation).

## 5. Constraints & Assumptions

### Constraints

| # | Constraint |
|---|-----------|
| C-01 | Плагин использует только PaperSpigot API 1.21. Никаких внешних зависимостей. |
| C-02 | Код флагов находится в модуле `comminusm`, пакет `ru.kyamshanov.comminusm`. |
| C-03 | Модель данных (таблицы `orders`, `work_fronts`) не содержит поля для хранения UUID armor stand или материала исходного опорного блока. UUID armor stand сохраняется в **PDC чанка** (`Chunk.getPersistentDataContainer`), а не в БД. Миграция схемы БД не требуется. Исходный материал опорного блока **не сохраняется** — при удалении флага блок заменяется на AIR. |
| C-04 | Armor stand создаётся через Bukkit API `world.spawn(location, ArmorStand::class.java)`. |
| C-05 | На сервере нет других плагинов (region protection, anti-grief). Защита флагов — исключительно ответственность ComminusmPlugin. |
| C-06 | Установка флагов разрешена **только в Overworld** (`world` по умолчанию). Список разрешённых миров задаётся в `config.yml` параметром `flag.allowedWorlds`. |

### Assumptions

| # | Assumption |
|---|-----------|
| A-01 | Между активацией флага и моментом обновления (загрузка чанка — lazy refresh, либо передача владения — синхронно) имя игрока на armor stand может устареть. Устаревшее имя — приемлемое поведение между обновлениями. |
| A-02 | BEDROCK является приемлемым материалом опоры для игрового дизайна (невозможно сломать в survival). Если нужно OBSIDIAN — настраивается в конфиге. |
| A-03 | Armor stand с `setMarker(true)` не создаёт помех при прохождении игрока (хитбокс отсутствует). |
| A-04 | Проверка 2 блоков воздуха над баннером достаточна для гарантии, что armor stand не будет заблокирован. |
| A-05 | Владелец флага определяется по UUID из БД. Никнейм резолвится через `Bukkit.getOfflinePlayer(uuid).name` на момент активации. |

## 6. Glossary

| Термин | Определение |
|--------|-------------|
| **Флаг ордера** | WHITE_BANNER, установленный в мире как центр зоны ордера. Содержит кастомное имя с ID ордера. |
| **Флаг трудового фронта** | RED_BANNER, установленный в мире как центр зоны трудового фронта. Содержит кастомное имя. |
| **Опорный блок** | Блок непосредственно под баннером, к которому баннер прикреплён. В текущей реализации — любой блок, поставленный игроком или сгенерированный миром. |
| **Неуничтожимый опорный блок** | Новый тип опорного блока — BEDROCK или OBSIDIAN, автоматически устанавливаемый при активации флага. Не может быть разрушен или сдвинут никаким игровым механизмом. |
| **Armor stand (стойка для брони)** | Vanilla Minecraft entity, используемая для отображения имени владельца над флагом. Невидимая, маркерная, без физики. |
| **Title (заголовок)** | Кастомное имя armor stand, видимое всем игрокам (аналог name tag). Содержит тип флага и никнейм владельца. |
| **Дюп флага** | Эксплойт: игрок получает больше флагов, чем разрешено системой (1 ордер + 1 фронт), через разрушение опорного блока, подбор дропа и запрос нового флага через `/party`. |
