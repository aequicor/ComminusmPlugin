# Design: Система приватов «Ордер и Фронт»

**Status:** Draft
**Date:** 2026-05-01
**Author:** @Main (orchestrator) / PO

## 1. Обзор

Система территориального управления в коммунистическом сеттинге. Два инструмента:

1. **Ордер на пользование** — личная жилплощадь с полной защитой (приват).
2. **Трудовой Фронт** — область приоритетной добычи ресурсов (шахта, лес, земля).

Вне Ордера или активного Фронта добыча блоков невозможна. Фронты могут пересекаться — кооперативная добыча.

## 2. Концепции

| Термин | Суть |
|--------|------|
| **Ордер** | Бюрократический документ Партии, дающий право *единоличного пользования* участком земли. 1 ордер на гражданина. |
| **Фронт работ** | Временный наряд на добычу ресурсов. Только 1 активный фронт. При пересоздании старый аннулируется. |
| **Трудодни** | Внутренняя валюта. Пассивный доход + сдача ресурсов в казну. Используется для прокачки Ордера. |

### Зависимости

- Paper API 1.21.11
- SQLite JDBC (встроен в JVM, не требует внешних библиотек)

## 4. Механика Ордера

### 4.1 Получение

1. Игрок вводит `/партия` → главное меню → кнопка «Получить Ордер».
2. `OrderService.create()` проверяет: нет других Ордеров в радиусе 30 блоков от игрока (пока нет флага — от позиции игрока).
3. Выдаётся **баннер с серпом и молотом** в инвентарь.
4. Запись в БД: `Order(ownerUUID, level=1, center=null, radius=2)`.
5. Игрок ставит баннер в мире → `BlockPlaceEvent` → `OrderService.activate(orderId, location)`:
   - Проверка: location не внутри чужого Ордера.
   - Сохраняется `center = location`, `radius = 2` (уровень 1 = 5×5).
   - PDC-метка на чанке для быстрой проверки.

### 4.2 Уровни и размеры

Радиус = расстояние от центра до края (half-width). Для квадрата размер = radius × 2 + 1.

| Уровень | Размер | Радиус | Стоимость (трудодни) |
|---------|--------|--------|---------------------|
| 1 | 5×5 | 2 | бесплатно |
| 2 | 7×7 | 3 | 30 |
| 3 | 9×9 | 4 | 80 |
| 4 | 11×11 | 5 | 150 |
| 5 | 15×15 | 7 | 300 |

### 4.3 Прокачка

1. ПКМ по флагу Ордера → `OrderMenu` → кнопка «Улучшить (N трудодней)».
2. `OrderService.upgrade(orderId)`:
   - Проверка: `currentLevel < 5`, `balance >= cost`.
   - `WorkdaysService.spend(uuid, cost)`.
   - `OrderRepository.updateLevel(orderId, newLevel)`.
   - Обновление PDC-метки чанка (радиус изменился).

### 4.4 Защита

- **Полная приватность:** только владелец может ломать и ставить блоки внутри Ордера.
- Чужие при попытке: сообщение «Чужая жилплощадь, товарищ! Обратитесь в партию за собственным Ордером.»
- Флаг неуничтожим никем кроме владельца (даже криперами/TNT).

### 4.5 Восстановление флага

Если флаг потерян (лава, баг): `/партия` → Ордер → «Восстановить флаг» → флаг возвращается на центр участка. Ордер не аннулируется.

## 5. Механика Фронта

### 5.1 Получение и активация

1. `/партия` → «Трудовой фронт» → выдаётся баннер фронта (другой узор/цвет).
2. Игрок ставит баннер в любом месте мира (шахта, лес, поле).
3. `BlockPlaceEvent` → `WorkFrontService.activate(playerUUID, location)`:
   - Если есть старый Фронт → деактивировать: удалить PDC-метку, удалить запись в БД.
   - Создать новый: `WorkFront(ownerUUID, center=location, radius=25)`.
   - PDC-метка на чанке.

### 5.2 Работа Фронта

- **Радиус:** всегда 25 блоков (50×50×50 куб).
- **Ограничение добычи:** игрок может добывать ресурсы ТОЛЬКО внутри своего активного Фронта или своего Ордера.
- **Кооперация:** Фронты разных игроков могут пересекаться. В зоне пересечения добыча разрешена всем владельцам пересекающихся Фронтов.
- **Вне Ордера/Фронта:** добыча запрещена → «Несанкционированная добыча ресурсов, товарищ! Получите Ордер или активируйте Трудовой Фронт.»

### 5.3 Пересоздание

Поставить новый флаг Фронта в другом месте = старый Фронт полностью удаляется.

## 6. Экономика (Трудодни)

### 6.1 Пассивный доход

- **1 трудодень за 10 минут онлайна.**
- Реализация: `PlayerListener` → Bukkit scheduler (каждые 10 минут проверять онлайн-игроков).

### 6.2 Сдача ресурсов в казну

- `/партия` → «Сдать ресурсы в казну» → `TreasuryMenu`.
- Игрок кладёт ресурсы в слоты GUI, нажимает «Сдать».
- Курсы (конфигурируемые):

| Ресурс | Трудодней за стак (64) |
|--------|------------------------|
| Булыжник | 4 |
| Уголь | 6 |
| Железо | 12 |
| Золото | 20 |
| Алмазы | 40 |
| Дерево | 4 |
| Земля | 1 |

## 7. Хранение данных

### 7.1 SQLite (основное)

- Файл: `plugins/ComminusmPlugin/data.db`
- WAL-режим для производительности.
- Таблицы:

```sql
CREATE TABLE IF NOT EXISTS orders (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    owner_uuid TEXT NOT NULL,
    level INTEGER NOT NULL DEFAULT 1,
    center_world TEXT,
    center_x INTEGER,
    center_y INTEGER,
    center_z INTEGER,
    radius INTEGER NOT NULL DEFAULT 3,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS work_fronts (
    owner_uuid TEXT PRIMARY KEY,
    center_world TEXT NOT NULL,
    center_x INTEGER NOT NULL,
    center_y INTEGER NOT NULL,
    center_z INTEGER NOT NULL,
    radius INTEGER NOT NULL DEFAULT 25,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS workdays (
    player_uuid TEXT PRIMARY KEY,
    balance INTEGER NOT NULL DEFAULT 0
);
```

### 7.2 PersistentDataContainer (PDC, оптимизация)

- На каждом чанке, содержащем центр Ордера или Фронта: PDC-тег с `order: <id>` или `front: <uuid>`.
- Используется для быстрой проверки при `BlockBreakEvent`/`BlockPlaceEvent` — сначала проверяем PDC чанка, затем БД.
- При загрузке чанка: сверка PDC с БД. Осиротевшие PDC-метки удаляются.

## 8. Потоки данных

### 8.1 Создание Ордера

```
Player → /партия → PartyMenu → "Получить Ордер"
  → OrderService.create()
    → OrderRepository.findNearby(playerLoc, 30) — проверка соседей
    → OrderRepository.insert(Order(level=1, radius=3))
    → Выдать баннер (серп и молот)
  → Игрок ставит баннер → BlockPlaceEvent
    → OrderService.activate(orderId, location)
      → Проверка: location не в чужом Ордере
      → OrderRepository.updateCenter(orderId, location)
      → Запись PDC-метки на чанке
```

### 8.2 Прокачка Ордера

```
Player → ПКМ по флагу → OrderMenu → "Улучшить (N)"
  → OrderService.upgrade(orderId)
    → Проверка: level < 5, balance >= cost
    → WorkdaysService.spend(uuid, cost)
    → OrderRepository.updateLevel(orderId, newLevel)
    → Обновить PDC-метку чанка
```

### 8.3 Создание Фронта

```
Player → /партия → PartyMenu → "Трудовой фронт" → FrontMenu
  → Выдать баннер фронта
  → Игрок ставит баннер → BlockPlaceEvent
    → WorkFrontService.activate(uuid, location)
      → Деактивировать старый Фронт (удалить PDC + БД)
      → WorkFrontRepository.upsert(WorkFront(uuid, location, radius=25))
      → PDC-метка на чанке
```

### 8.4 Добыча блоков

```
Player ломает блок → BlockBreakEvent
  → BlockListener:
    1. Блок в Ордере игрока? → РАЗРЕШИТЬ
    2. Блок в чужом Ордере? → ЗАПРЕТИТЬ ("Чужая жилплощадь, товарищ!")
    3. Блок в активном Фронте игрока? → РАЗРЕШИТЬ
    4. Блок в чужом Фронте? → РАЗРЕШИТЬ (кооперация)
    5. Блок вне всех зон? → ЗАПРЕТИТЬ ("Несанкционированная добыча!")
```

## 11. Обработка ошибок и краевые случаи

| Ситуация | Поведение |
|----------|-----------|
| Добыча без Ордера/Фронта | Запрет. «Товарищ, добыча без Ордера или Наряда — вредительство!» |
| Флаг Ордера внутри чужого | Запрет. Флаг возвращается в инвентарь. «Данная территория уже распределена партией.» |
| Флаг Ордера < 30 блоков от чужого | Запрет. «Слишком близко к соседскому наделу. Партия требует отступ в 30 блоков.» |
| Потеря флага Ордера | Восстановление через меню — флаг на центр участка. |
| Смена никнейма | Привязка к UUID — не теряется. |
| Краш при записи в БД | WAL + транзакции. PRAGMA integrity_check при старте. |
| Осиротевшая PDC-метка | Сверка при загрузке чанка → удаление. |
| Нехватка трудодней | Показ баланса и недостающей суммы. |
| Уровень 5 (макс) | Кнопка «Улучшить» скрыта. |
| Два игрока с пересекающимися Фронтами | Оба могут копать в зоне пересечения. |

## 12. Тестирование

### 12.1 Модульные тесты (JUnit 5)

- `OrderService`: математика пересечений, расчёт радиуса по уровню, валидация.
- `WorkFrontService`: логика «только 1 активный».
- `WorkdaysService`: начисление, списание, достаточность баланса.
- `GuiUtils`: генерация ItemStack, форматирование lore.
- `PluginConfig`: парсинг конфига, значения по умолчанию.

### 12.2 Интеграционные тесты (MockBukkit)

- Полный флоу создания Ордера: команда → меню → БД → баннер.
- Прокачка: ПКМ → меню → списание → обновление уровня.
- Защита: `BlockBreakEvent` на чужом Ордере → cancel.
- Кооперативная добыча: два игрока с пересекающимися Фронтами.
- Пересоздание Фронта: новый флаг → старый деактивирован.

### 12.3 Ручное тестирование (run-paper)

- Полный цикл: создать Ордер → прокачать до 5 → проверить защиту.
- Стресс-тест: 50 игроков × Ордер + Фронт.
- Краш-тест: убить сервер при записи в БД → перезапустить → целостность.

### 4.6 Правила размещения

- **Минимальное расстояние между центрами Ордеров:** 30 блоков (измеряется от центра до центра).
- **Флаг внутри чужого Ордера:** запрещено при активации.
- **Флаг ближе 30 блоков к центру чужого Ордера:** запрещено при активации.
- Ордера не пересекаются никогда (гарантируется математикой + проверкой).

## 15. Допущения и риски

1. **Производительность:** проверка каждого `BlockBreakEvent` через SQLite может создать нагрузку при 50+ игроках. Решение: PDC-кэш на чанках + in-memory кэш активных Ордеров/Фронтов в `OrderService`/`WorkFrontService`.
2. **Чанковая гранулярность PDC:** на границе чанков может быть задержка. Решение: проверять соседние чанки при `BlockBreakEvent`.
3. **Разрушение флага TNT/крипером:** требует отдельной обработки в `EntityExplodeEvent` и `BlockExplodeEvent`.
